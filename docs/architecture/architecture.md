# Архитектура ACCAC

## Назначение

ACCAC предназначен для визуализации состава и состояния антенных комплексов. Настольный клиент получает данные из PostgreSQL и строит интерфейсный макет на основе предметных сущностей и описания UI-компонентов, хранящихся в БД.

## Основные компоненты

- `src/forms/` — формы Lazarus для главного окна, редактора данных и меню.
- `src/domain/` — логика работы с сущностями предметной области: центры, города, антенны, модели, статусы, макеты и UI-компоненты.
- `src/infrastructure/unitdb.pas` — подключение к PostgreSQL и чтение конфигурации.
- `db/postgresql/migrations/` — схема, таблицы, индексы, функции, процедуры, триггеры, начальные данные и grants.
- `db/postgresql/fixtures/hex/` — данные для наполнения бинарных полей, используемых макетом и моделями.

## Архитектурная схема

Система реализована как двухзвенная архитектура:

1. Клиент на Lazarus отображает макет, данные по антеннам и предоставляет CRUD-интерфейс редактора.
2. PostgreSQL хранит как доменные данные, так и структуру визуального макета.

Отдельного серверного API-слоя в репозитории нет: клиент работает напрямую с БД через `TPQConnection`, `TSQLTransaction` и `TSQLQuery`.

## Ключевые сущности

- `tb_CTR` — центры.
- `tb_CITY` — города.
- `tb_MDL` — модели антенн и связанные изображения.
- `tb_ANT` — антенны.
- `tb_STAT` — статусы антенн.
- `tb_LYT` — макеты.
- `tb_CMP` — UI-компоненты, определяющие состав и размещение элементов на макете.

## Поток данных

1. Клиент читает локальный `accac.ini` из корня репозитория, рядом с бинарником, из `build/config/` или из `src/config/`.
2. `unitdb.pas` устанавливает соединение с PostgreSQL.
3. Главная форма загружает активный макет и связанные компоненты из `tb_CMP`.
4. Для элементов антенн подгружаются связанные модели, города и статусы.
5. Редактор позволяет изменять сущности и сразу обновлять отображение.
